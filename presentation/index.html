ed (ir-)regular expressions

  • Any character except ".", "[", "]", "^", "$", or "\", matches itself.
  • Any backslash-escaped character except for `{', `}', `(', `)', `<' and `>', matches itself.
  • "." matches any single character.
  • [chars] matches any of the listed chars (also takes ranges with "-", negation with "^")
  • [:alnum:], [:alpha:], etc.
  • "^" as the first character anchors the regular expression to the beginning of a line.
  • "$" as the last character anchors the regular expression to the end of a line.
  • \(capture\)
  • \digit is a backreference.
  • "*" matches the preceding single character term zero or more times.
  • \{n,m\} or \{n,\} or \{n\} range-limited repitition for single character terms

The plan of attack, gritty details

  • Data is represented as text stored in text files
    • Scalars can be represented by a line consisting of the natural string representation of the value
    • One dimensional arrays can be represented by multiline files with the first line containing the first element, the second line thesecond element, and so forth
    • Hashes can be represented by text files with an even number of lines, representing key-value pairs
    • Data structures can be nested by replacing all "\"s with "\\" and joining the lines with "\"-delemiters
  • Code is represented as text stored in text files
    • The main program will consist of commands which we'll pipe to ed
    • It can create variables by building text representing data in the buffer and writing it to temporary files with names like x.int or goo.array-of-float
    • It can create sub-routines by building text representing code in the buffer and writing it to temporary files with names like my-function.ed
    • It will need a way to produce lines consisting of a single "." in the output

Taking it a step farther

Despite my whining, ed was overkill.

We could have done anything we wanted with just string replacement & recursion.

For example, consider the rules:

  • s/#0/0##/
  • s/1/0#/
  • s/0//

If we start with the string 1100 (=0C hex, or 12 decimal) and do leftmost matches first we have:

1100 → 0#100 → #100 → #0#00 → 0###00 → ###00
→ ##0##0 → #0####0 → 0######0 → ######0 →
...and ultimately... → ############

From this sort of thing we could build up to integer math and...well, anything. Substitution systems are Turing complete.

Well yeah, but...

Taking it a step farther

In config files you often find things like: 

install_dir = /usr/local/foo/
lib_dir     = /usr/local/foo/lib/
config_dir  = /usr/local/foo/config/
log_dir     = /usr/local/foo/log/

...where the semantic intent is more like: 

install_dir = /usr/local/foo/
lib_dir     = install_dir/lib/
config_dir  = install_dir/config/
log_dir     = install_dir/log/

Taking it a step farther

There are cases where we might write something like: 

virtual_host_preamble `choose_your_mate.com' virtual_host_body
virtual_host_preamble `online_dating.org' virtual_host_body
virtual_host_preamble `you_know_its_time.edu' virtual_host_body
virtual_host_preamble `matchorama.com' virtual_host_body

but we really want to write something more like: 

virtual_host(`choose_your_mate.com')
virtual_host(`online_dating.org')
virtual_host(`you_know_its_time.edu')
virtual_host(`matchorama.com')

Taking it a step farther

There are of course a few low-hanging fruit "nice to have" features:

  • A (nestable) way to guard strings from replacement
  • A way to define new replacements as you go (getting rid of the outer recursion
  • A way to add comments (since these things could quickly get quite cryptic)
  • A way to do write conditionals

These are "nice to have" but not essential, as they could all be synthisized if they weren't provided.
